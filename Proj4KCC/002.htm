Настройка уведомлений о сигналах по электронной почте

 

Данный документ описывает шаги по настройке почтовых уведомлений от сигнальной системы РСДУ.

Почтовые сообщения позволяют уведомлять пользователей о важных событиях комплекса. К примеру, администраторы комплекса могут себе настроить уведомления о сбоях в работе ПО.

Способ доставки почтовых уведомлений

За обработку сигналов в комплексе РСДУ отвечает серверный модуль «Сервер сигнальной системы» (ssbsd).

Сервер сигнальной системы при получении сигнала проверяет, подписан ли кто-то из пользователей на получение уведомления по почте. Если такие пользователи имеются, сервер сигнальной системы подключается к настроенному SMTP-серверу и отправляет соответствующее письмо.

Далее письмо доставляется по протоколам обмена электронной почтой (SMTP) согласно настройкам почтового сервера.

Как правило, настраивается локальный (для РСДУ) SMTP-сервер, предназначенный для отправки сообщений по внутренней сети и перенаправляющий сообщения к уже существующему почтовому серверу в технологической сети предприятия. Данное руководство описывает именно такой способ настройки локального почтового сервера.

Необходимые условия

На АРМ, с которого работает администратор, должен быть установлен клиент РСДУ и запущено приложение Навигатор БД.

У администратора должны быть следующие права:

·         Базовая информация комплекса – «Чтение», «Настройка подсистемы», «Вход в систему ОИК».

·         Подсистема контроля и резервирования – «Чтение», «Настройка подсистемы».

На АРМ должен быть установлен клиент ssh (например, PuTTY) для доступа к серверам, на которых работает Сервер сигнальной системы (ssbsd).

Сигнальная подсистема комплекса РСДУ должна быть настроена в соответствии документом «Настройка сигнальной системы. Руководство пользователя».

Обеспечен доступ с серверов РСДУ к настроенному почтовому серверу в технологической сети предприятия.

 

Необходимые знания

1.     IP-адрес основного и резервного сервера, на котором работает Сервер сигнальной системы (ssbsd).

2.     Пароль суперпользователя (root) на указанные сервера.

Предварительные проверки

Проверка Сервера сигнальной системы (ssbsd)

Подключитесь к серверу, на котором находится Сервер сигнальной системы по ssh от имени суперпользователя(root). Для этого можно использовать программу PuTTY. Перед вами появится командная строка.

Для отправки почтовых сообщений необходимо, чтобы серверный модуль signal3(ssbsd) был собран с опцией SSBSD_SEND_MAIL_PRESENT.

Проверить это можно, введя в командную строку следующую команду:

grep "Send mail option" /var/log/ema/ssbsd/ssbsd.log

 

У команды должен быть непустой вывод, вы должны увидеть строки вида:

14.09.2018 11:14:47.768 [INFO ] Send mail option is Enabled.

 

Если предыдущая команда не вывела такую строку, то значит сервер сигнальной системы скомпилирован без поддержки этой функции. Для ее включения необходимо пересобрать сервер сигнальной системы с этой опцией в соответствии с документом «Сборка и установка серверного ПО РСДУ. Руководство администратора».

В данном руководстве обратите внимание на следующие разделы:

·         Приложение 2. Список дополнительных объявлений define для сборки проектов

·         Раздел «Дополнительные возможности скрипта configure  «ключ  --defines-<Project Name>=Define1,Define2».

Установка и настройка локального почтового сервера

Для резервированных полукомплектов (ОИК1/ОИК2) необходимо выполнить установку и настройку для обоих ОИК.

Установка postfix

Подключите установочный диск SLES к серверу.

Пакет postfix находится на установочном диске SLES и по умолчанию ОС считает диск репозиторием. Для того, чтобы установить postfix, необходимо в командной строке ввести команду:

zypper install postfix

 

Во время установки программа спросит, действительно ли вы хотите установить пакет, на этот вопрос нужно ответить «y», или «yes», или просто нажать ENTER, так как это вариант по умолчанию.

Настройка почтового сервера, работа с конфигурационным файлом

Postfix, как правило, настраивают для пересылки сообщений от имени учётной записи на почтовом сервере, который находится в технологической сети предприятия. В этой главе описывается именно такой способ настройки.

Для настройки параметров веб-сервера используется конфигурационный файл /etc/postfix/main.cf.

Для отправки писем от имени другой учётной записи локальному postfix нужны логин и пароль от этой учётной записи на почтовом сервере в технологической сети.

Установить необходимые логин и пароль:

1.       Создать файл /etc/postfix/relay_passwd со следующим содержимым: имя_почтового_домена имя_пользователя:пароль

                Например:

ema.ru rsdu_notification@ema.ru:passwordforrsdu

2.       Сгенерировать на основе этого файла базу данных, которую понимает postfix, используя следующую команду:

               postmap /etc/postfix/relay_passwd

                Должен появиться файл /etc/postfix/relay_passwd.db.

                А файл /etc/postfix/relay_passwd после этого можно удалить.

3.       В конец файла /etc/postfix/main.cf с помощью текстового редактора нужно добавить следующие строки:

myhostname = mail.rsdu.local
relayhost = [mail.yournetwork.ru]
mynetworks = 0.0.0.0/0
inet_interfaces = all
smtp_sasl_auth_enable = yes
smtp_sasl_password_maps = hash:/etc/postfix/relay_passwd
smtp_sasl_security_options = noanonymous
smtp_sasl_mechanism_filter = login

Установить значение для myhostname, совпадающее с выводом следующей команды:

hostname –f

Установить значение для relayhost для вашей сети, где yournetwork – имя вашей сети, например:

relayhost = [mail.ema.ru]

Остальные параметры оставить без изменения.

Внимание!!! При настройке ретрансляции необходимо, чтобы сервер электронной почты смог подключиться к postfix по данному доменному имени. Для этого на "пересылающем"(relay) сервере нужно правильно сконфигурировать DNS или добавить myhostname из main.cf в файл /etc/hosts на почтовом сервере в сети предприятия.

После этого перезапустить модуль ssbsd. Посмотреть лог, если MyDomainName указан (none), то проверить, что в сетевых настройках сервера прописано имя домена, и перегрузить хост.

Проверка работоспособности

Для того, чтобы настройки применились необходимо запустить или перезапустить, если уже запущен, сервер postfix. В командной строке нужно ввести:

postfix status

 

Если сервер выключен, команда выведет:

postfix/postfix-script: the Postfix mail system is not running

 

В таком случае его нужно запустить командой:

postfix start

 

Если команда «postfix status» вывела следующее, значит сервер запущен:

postfix/postfix-script: the Postfix mail system is running: PID: 23835

 

В таком случае нужно попросить postfix перечитать конфигурационный файл командой:

postfix reload

 

После этого еще раз проверяем статус сервера командой postfix status, и вывод теперь должен быть таким:

postfix/postfix-script: the Postfix mail system is running: PID: 23835

 

Если вывод отличается, скорее всего в конфигурационном файле содержится ошибка.

Проверка автозагрузки

Узнать версию ОС

В разных версиях SLES проверка и редактирование автозагрузки выполняется по-разному. Узнайте версию SLES введя в командную строку команду:

cat /etc/issue

 

Вывод будет содержать версию ОС:

Welcome to SUSE Linux Enterprise Server 10 SP4  (x86_64) - Kernel \r (\l).

 

SLES 10, SLES 11

Для этих версий действия одинаковые:

chkconfig postfix

 

Вывод команды должен быть «postfix  on», если это не так, то postfix нужно добавить в автозагрузку командой:

chkconfig postfix on

 

После этого нужно заново проверить добавился ли почтовый сервер в автозагрузку.

SLES 12

Для проверки добавлен ли почтовый сервер в автозагрузку, введите в командную строку следующую команду:

 systemctl status postfix

 

В выводе программы обратите внимание на строку Loaded, в ней после имени запускаемого сервиса должно быть enabled:

postfix.service - Postfix Mail Transport Agent

   Loaded: loaded (/usr/lib/systemd/system/postfix.service; enabled; vendor preset: enabled)

   Active: active (running) since Вт 2018-09-25 17:19:51 +07; 20h ago

  Process: 1605 ExecStartPost=/etc/postfix/system/cond_slp register (code=exited, status=0/SUCCESS)

 

...

 

сен 25 17:19:51 sles12sp3 systemd[1]: Started Postfix Mail Transport Agent.

Hint: Some lines were ellipsized, use -l to show in full.

 

Если в строке написано disabled, то нужно добавить в автозагрузку почтовый сервер следующей командой:

systemctl enable postfix.service

 

После добавления в автозагрузку снова проверить командой systemctl status postfix.

 

Настройка переменных postfix в РСДУ

Чтобы сервер сигнальной системы знал, куда отправлять письма, необходимо задать адрес SMTP сервера у сервера сигнальной системы в Навигаторе БД РСДУ. Для этого в свойствах сервера сигнальной системы выбрать вкладку "Инициализационные параметры". Добавить в список параметр SMTP_HOST_NAME, при желании можно использовать несколько серверов: для этого есть опции SMTP_HOST_NAME2 и SMTP_HOST_NAME3 (более подробно см. документ «Картридж «Серверы РСДУ»).

Рисунок 1 – Окно добавления параметра

 

Если postfix настраивается на той же машине, где работает сервер сигнальной системы, в качестве адреса можно указать 127.0.0.1.

 

Настройка уведомлений для конкретного пользователя

Для получения данных по электронной почте необходимо в Навигаторе в свойствах конкретного пользователя добавить email-адрес пользователя (подробнее см. «Картридж «Пользователи и процессы»).

Выбор конкретных сигналов

В свойствах пользователя необходимо указать сигналы, о которых нужно уведомлять по почте. Для этого на вкладке «Свойства сигналов» для нужных сигналов установить флаг "Уведомлять о событии на почтовый адрес" (подробнее см. «Картридж «Пользователи и процессы»).

В данной инструкции опущены шаги по настройке зон ответственности и групп сигналов. Если требуется данная настройка, то см. «Настройка сигнальной системы. Руководство пользователя».

Шаблоны для сообщений по электронной почте можно настроить, используя руководство «Картридж «События и сообщения».

Применение изменений

После обновления настроек в Навигаторе БД, необходимо перезапустить Сервер сигнальной системы (ssbsd) на ОИК1 и ОИК2 (см. «Управление комплексом реального времени» или «Сопровождение серверного ПО для ОС Linux»).
