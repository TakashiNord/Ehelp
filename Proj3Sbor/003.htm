1 Описание оборудования и точек доступа в комплексе РСДУ5

Документ описывает последовательность заполнения раздела описания серверов и точек доступа, рекомендации и указания по распределению серверов по хостам.

Документ содержит инструкции для описания:

ü  кластера реального времени (РВ);

ü  хоста;

ü  сетевого интерфейса;

ü  технологического сервера;

ü  сервера данных и сервера доступа;

ü  точек доступа;

ü  коммуникационной платы;

ü  блока адаптеров ТМ.

Примечание. Все сервера РВ работают под ОС Linux, за исключением серверов сбора/передачи по OPC/DA и сервера сбора OPC/HDA (ОС Windows).

Описание оборудования и точек доступа хранится в разделе AD_ базы данных технической информации (БДТИ) комплекса РСДУ.

В навигаторе БД существует узел «Оборудование и точки доступа».

Рисунок 1 – Узел «Оборудование и точки доступа»

В БДТИ должны быть описаны следующие сущности:

ü  сервер (хост или кластер) БДТИ;

ü  резервный сервер БДТИ, если входит в конфигурацию;

ü  сервер архивов, если входит в конфигурацию;

ü  используемые адаптеры, платы и порты (MOXA, BITBUS,..);

ü  многомашинная вычислительная система (МВС) – кластер БДРВ №1 с описанием хоста;

ü  резервная МВС – кластер БДРВ №2 с описанием хоста;

ü  все сервера (модулей) комплекса и их точки доступа для каждого описанного хоста комплекса;

ü  интеграционная платформа (при наличии);

ü  основной и резервный сервер отображения (при наличии);

ü  сервер ресурсов (файловый сервер).

1.1 Необходимые условия

На машине, с которой работает администратор, должен быть установлен клиент РСДУ и запущено приложение Навигатор БД.

У администратора должно быть право на чтение и изменение «Подсистема контроля и резервирования», чтение на подсистему «Базовая информация комплекса».

1.2 Необходимые знания

Перед чтением данного руководства рекомендуем изучить следующие материалы:

ü  «Картридж "Точки доступа". Руководство администратора»;

ü  «Картридж "Комплекс автоматики". Руководство администратора»;

ü  «Картридж "Серверы РСДУ". Руководство администратора»;

ü  «Картридж "Пользователи и процессы". Руководство администратора»;

ü  «Структура базы данных. Раздел “Комплекс автоматики”. Руководство администратора».

 

2 Описание многомашинной вычислительной системы

1.    Создать узел, являющийся родительским для всех хостов, входящих в МВС. Тип узла – «Многомашинная вычислительная система».

Рисунок 2 – Выбор типа узла

2.    Определить модель МВС – «Кластер Industrial PC».

Рисунок 3 – Выбор модели МВС

3.    Определить свойства RTDB (БДРВ): выбрать необходимый тип сетевого интерфейса, задать число хостов в кластере создаваемой системы.  Максимально возможное число хостов для кластера Industrial PC задать равным числу хостов.

Рисунок 4 – Задание свойств кластера БДРВ

4.    Нажать кнопку «ОК».

5.    Если в Навигаторе в дереве появился узел типа МВС, это означает, что описание МВС существует в БДТИ.

2.1 Описание хоста (слота) комплекса

Примечание. Независимо от положения в иерархии (внутри кластера находится хост или является отдельным), описание хоста комплекса выполняется одинаково.

2.1.1 Узел хоста

1.    Создать новый узел, выбрав тип оборудования – «Технологическая PC».

2.    Задать имя и краткое наименование хоста. Удобно в качестве имени указывать DNS имя или его часть. Например, «ОИК №1».

3.    Выбрать модель хоста. При отсутствии в списке необходимой модели можно выбрать общий тип. Например, «Сервер для реального времени».

4.    Задать номер хоста, выбрав уникальные идентификаторы во множестве хостов кластера.

5.    Если в дереве Навигатора БД появился узел с типом «Технологическая PC», значит хост описан в БДТИ.

2.1.2 Сетевые интерфейсы хоста

Примечание. Хост может иметь подключение во внешнюю сеть, предназначенную для доступа к хосту клиентов комплекса, и во внутреннюю (технологическую) сеть, предназначенную только для обмена между технологическими серверами комплекса (на коммуникационных серверах вместо внутренней сети описывается сеть приборов). Количество подключений не ограничено, но, как правило, используются указанные два. За каждым подключением закрепляется сетевая карта. При описании хоста, не входящего в МВС, внутренняя сеть не нужна.

1.    Для созданного узла хоста создать внешнюю сеть, выбрав тип оборудования – «Внешняя сетевая карта».

2.    Выбрать модель карты. Например, «Сетевая карта INTEL».

3.    На вкладке «Свойства сетевой карты» определить используемые адреса: адрес Ethernet для внешней сети задавать необязательно (поскольку обмен идет по TCP/IP), поставить метку «Есть IP-протокол», задать необходимый TCP/IP адрес, имя домена, сетевую маску.

Рисунок 5 – Свойства сетевой карты

4.    Для созданного узла хоста создать внутреннюю сеть: для кластера адрес Ethernet для внутренней сети задавать необязательно (поскольку обмен идет по UDP), поставить метку «Есть IP-протокол», задать необходимый TCP/IP адрес, сетевую маску.

5.    Если в дереве Навигатора появился узел с типом «Внешняя сетевая карта», это значит, что сетевая карта описана в БДТИ.

2.1.3 Описание серверов

Для созданного узла хоста необходимо описать выполняющиеся на хосте приложения (серверы), следуя пунктам, указанным ниже.

Примечание. Рекомендуемое и отчасти обязательное распределение приложений по хостам представлено в Приложение Д.

1.    Создать узел с типом «Группа серверов», назвав его, например: «Приложения». Узел будет родительским для всех описанных приложений на хосте.

Рисунок 6 – Узел «Приложения»

2.    Описать выполняющиеся на хосте приложения, следуя пунктам «Описание технологического сервера, сервера доступа и сервера данных».

2.1.3.1 Описание технологического сервера

Ниже описан пример для сервера коммутационных аппаратов.

1.    Выбрать тип сервера и название модуля сервера, который собираетесь описать, в соответствии с Приложение А (если нужный загружаемый модуль не найден, обратиться к разработчикам комплекса). Нажать кнопку «Далее».

Рисунок 7 – Выбор типа сервера

2.    По желанию изменить имя и краткое наименование сервера.

Примечание. Желательно, чтобы наименование приложения совпадало с именем пользователя (меньше путаницы). Наименование приложения и, соответственно, имя пользователя МВС удобно дополнять номером хоста (слота), на котором оно выполняется, и номером БДРВ (при наличии резервного комплекса). Именно по такому правилу автоматически формируется имя, например: «Сервер силовых коммутационных аппаратов хост 0 Кластер БДРВ №1», краткое наименование (название приложения + номер БДРВ + номер хоста), например: pwswtd10.

внимание!

При описании серверов данных краткое наименование обязательно должно иметь своей частью (первой) наименование загружаемого модуля.

 

Рисунок 8 – Базовые параметры приложения

3.    Нажать кнопку «Далее».

4.    Выбрать таблицу – список обрабатываемых приложением параметров в соответствии с Приложение А.

Рисунок 9 – Параметры приложения

5.    Задать ini-параметры для сервера, в соответствии с Приложение Б.

Рисунок 10 – INI-параметры приложения

6.    Задать в соответствии с Приложение В точки доступа.

Рисунок 11 – Задание точек доступа

7.    Нажать кнопку «Готово».

8.    Если в дереве появился узел с типом «Технологический сервер», это значит, что сервер описан в БДТИ.

2.1.3.2 Описание точек доступа сервера

Примечание.  Данный пункт следует выполнить, если по умолчанию при создании сервера какие-то точки доступа не создались или необходимо изменить описание существующей точки доступа.

1.    Задать название, краткое наименование точки доступа.

Рисунок 12 – Общие свойства точки доступа

2.    Выбрать тип точки доступа, нажав на кнопку «Сменить тип…».

3.    Нажать на кнопку «Далее».

4.    Привязать точку доступа к своему сервисному порту (сервисы описаны в таблице AD_SERVICE), выбрать протокол и интерфейс в соответствии с Приложение В.

Примечание. Для обмена по внутренней (технологической) сети используется протокол UDP. Для обмена по внешней сети: TCP, UDP (прикладной протокол разработки «ЭМА», функционирующий поверх коммуникационного протокола UDP и обеспечивающий гарантированную доставку и передачу пакетов неограниченного размера), UDP Clear (коммуникационный протокол UDP со стандартными для него ограничениями).

Рисунок 13 – Свойства точки доступа

5.    Нажать на кнопку «Готово».

6.    Если точка доступа появилась в списке в Навигаторе, это означает, что она описана в БДТИ.

2.1.3.3 Описание сервера доступа и серверов данных

Примечание 1. Сервер доступа может иметь привязанные к нему серверы данных, которые загружаются как любые другие технологические серверы.

Примечание 2. Сервер доступа может не иметь серверов данных. В этом случае он используется для предоставления информации о статусе своего хоста (основной/резервный).

1.    Следовать пунктам «Описание технологического сервера», «Описание точек доступа сервера», учитывая, что сервера данных описываются как дочерние узлы для своего сервера доступа.

Рисунок 14 – Описание сервера данных

2.    При настройке точки доступа для сервера данных необходимо определить сервисы, к которым он будет предоставлять доступ, и тип данных, обрабатываемый данным сервисом. Для этого нажать на кнопку «Свойства сервиса» в окне настройки точки доступа (кнопка активируется только для приложений с типом «Сервер данных»), выбрать поддерживаемые типы в соответствие с Приложение Г.

Рисунок 15 – Выбор типов данных для сервера данных

Если точка доступа появилась в списке в Навигаторе БД, это означает, что она описана в БДТИ.

CHANNEL_ALIVE_TIMEOUT

N

Проверка активности канала (в сек.):

0 – проверка активности канала отключена;

N>0 – включена проверка активности канала (проверяется, был ли за последние N секунд обмен по каналу (и прием, и передача)). Если за этот промежуток времени либо прием, либо отправка не срабатывали, тогда закроется текущее соединение. За этим последует пересоздание драйвера и соответствующего соединения.

Достоверизация, расхождение в секундах.  

CHECK_TIME

10

Период проверки (в секундах).

Используется многими серверами как период проверки состояния устройств, сервисов, пользовательских сессий.

Для сервера сбора с УТМ - период контроля адаптеров ТМ в случае применения двух корзин с адаптерами ТМ (одна корзина для основного сервере, вторая - для резервного). В случае отказа (сбоя) одного из адаптеров ТМ происходит переход на резервный сервер с резервной корзиной адаптеров.

Для отключения контроля необходимо задать CHECK_TIME = 0. Это необходимо сделать, если применяется одна корзина с адаптерами ТМ (без резерва).

Приложение Д. Распределение модулей РВ по хостам

Д.1 Правила для распределения модулей РВ

Сервер записи архивов должен быть описан на всех хостах, на которых находятся технологические сервера, архивы которых должны быть записаны.

Если коммуникационный сервер сбора работает на выделенном узле для сбора/передачи данных, у коммуникационного сервера сбора и передачи данных, расположенного на выделенном узле, должна быть описана точка доступа OIC-TCP.

На отдельно выделенном хосте (не входящем в кластер) должны быть обязательно описаны:

ü  агент системного монитора;

ü  сервер сигнальной системы;

ü  сервер доступа;

ü  мост для взаимодействия комплексов;

ü  системный монитор;

ü  сервер записи архивов (если это хост, осуществляющий сбор).

Д.2 Рекомендации по распределению модулей РВ

Следующие модули РВ, как правило, располагаются на выделенном хосте, который входит в состав МВС:

ü  коммуникационный сервис сбора и передачи данных;

ü  управление ДЩ;

ü  сервер сбора УТМ.
