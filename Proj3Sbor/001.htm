Настройка коммуникационного сервиса сбора и передачи данных РСДУ5

 

Глоссарий

КП – контролируемый пункт.

ПУ – пункт управления.

КС – коммуникационный сервис сбора и передачи данных РСДУ5.

Общее описание

Коммуникационный сервис сбора и передачи данных РСДУ5 обеспечивает обмен по протоколам МЭК870-5-101, МЭК870-5-103, МЭК870-5-104 (далее - МЭК), FDST, реализуя функции ПУ (master) и КП (slave); для протокола МЭК870-5-101 поддерживается также небалансный режим обмена в канале.

Дополнительно в сервере сбора обеспечивается обмен по промышленному протоколу MODBUS с реализацией функций ПУ (master) и КП (slave).

Кроме этого, в сервере реализованы протоколы опроса измерителей ION, СПЕ, СЭТ, ИРТ, ТЭКОН.

Необходимые условия

Перед чтением данного руководства рекомендуется изучить следующие материалы:

ü  Руководство администратора. Справочная информация по распределенной системе сбора данных.

ü  Руководство администратора. Описание параметров в подсистеме сбора и передачи РСДУ.

ü  Руководство администратора. Описание структуры БД РСДУ. Таблицы распределенной подсистемы сбора данных.

ü  Руководство администратора. Создание нового сегмента сбора в разделе DA.

Крайне рекомендуется заранее ознакомиться с описанием профиля протокола, для которого необходимо выполнить конфигурацию обмена. Например, «УСТРОЙСТВА И СИСТЕМЫ ТЕЛЕМЕХАНИКИ. Часть 5. Протоколы передачи. Раздел 101. Обобщающий стандарт по основным функциям телемеханики» п.8 (для МЭК870-5-101) или «УСТРОЙСТВА И СИСТЕМЫ ТЕЛЕМЕХАНИКИ. Часть 5. Протоколы передачи. Раздел 104. Доступ к сети для ГОСТ Р МЭК 870-5-101 с использованием стандартных транспортных профилей»  п. 9 (для МЭК870-5-104).

Необходимые знания

Описание сбора/передачи для МЭК870-5-101(104)

Независимо от выполняемых функций (КП или ПУ) для МЭК870-5-101(104) необходимо:

1.     Согласовать профили протокола.

2.     Определить канальный адрес (Link address) КП (только для МЭК870-5-101).

3.     Определить общий адрес ASDU (Common address of ASDU). Общий адрес ASDU определяет объект, содержащий список стандартных типов ASDU. Таких объектов может быть несколько с различными адресами.

4.     Определить стандартные типы ASDU (Standard ASDU), которые будем использовать для передачи необходимых параметров.

5.     Согласовать адресацию информационных объектов внутри стандартного типа ASDU. Адресация может быть сквозной, когда адрес объекта информации (Information object address) возрастает на единицу для каждого следующего объекта информации. Также адресация может быть индивидуальной для каждого типа.

6.     При задании ширины поля «Причина передачи» (Cause of transmission), равной 2 байта, необходимо определить собственный адрес станции (Originator address) (обязательно для МЭК870-5-104).

7.     Согласовать периодичность генерации ПУ команд:

ü  общего опроса (запрос с КП  значений ТИ, ТС);

ü  опроса счетчиков (запрос с КП интегральных параметров);

ü  синхронизации времени.

8.     Если требуется, определить для КП периодичность генерации  циклических обновлений параметров на ПУ. Можно не определять, если ПУ будет с необходимым периодом вести общий опрос.

Информация о согласовании пунктов 1-8 должна быть известна как на КП, так и на ПУ.

При выполнении пунктов 7,8 учитывать, что при изменении ТС на КП будет сгенерировано спорадическое сообщение с информацией о новом состоянии и подготовлено к немедленной отправке на ПУ. Спорадическая передача по изменению ТИ возможна только при запрете циклической передачи параметров.

Описание сбора/передачи для FDST

Независимо от выполняемых функций (КП или ПУ) для FDST необходимо:

1.     Определить собственный адрес станции (OWNER) и удаленной станции (WHOM) адрес.

2.     Согласовать адресацию передаваемых параметров для каждого передаваемого типа (ТИ, ТС, ЦБИ).

3.     Если требуется, определить для КП периодичность генерации  циклических обновлений параметров на ПУ. Спорадическая передача включается при запрете циклической передачи параметров.

Настройка сегмента распределенной подсистемы сбора

Коммуникационный сервер в РСДУ5 привязывается к одному из сегментов распределенной подсистемы сбора.

Управляющая ЭВМ

Для начала создаётся управляющая ЭВМ.

В Навигаторе Базы данных РСДУ5 («DBView») в разделе «Комплекс автоматики», в каталоге «Распределенная система сбора» необходимо добавить новую систему сбора. Этот процесс описан в документе «Создание нового сегмента сбора в разделе DA. Руководство администратора».

Номер машины различен для каждого сегмента.

Управляющая ЭВМ может быть выведена из работы включением опции «Выведен из работы».

Магистральный интерфейс

Для созданного узла управляющей ЭВМ создаются узлы магистральных интерфейсов. Коммуникационный сервис поддерживает следующие интерфейсы:

ü Последовательный интерфейс;

ü Ethernet интерфейс.

Интерфейс определяется полем «Тип интерфейса» и может быть:

ü «Сетевые карты Ethernet»;

ü «Стандартный COM-порт IBM PC»;

ü «Порты ETHERGATE»;

ü «Модем поверх TCP»;

ü «Драйвер MOXA TTY»;

ü «Плата MOXA C168H/PCI».

Поля «Базовый адрес», «Номер прерывания» и «Номер канала прямого доступа к памяти интерфейса» в настоящее время не используются.

Интерфейс может быть выведен из работы включением опции «Выведен из работы».

Коммуникационный порт

Для созданного узла магистрального интерфейса необходимо добавить узлы коммуникационных портов.

Для порта в первую очередь необходимо определить используемый протокол обмена.

Номер коммуникационного порта для последовательного интерфейса определяет физический порт (1 соответствует COM1 и т.д.), для интерфейса Ethernet – TCP порт.

Для последовательного интерфейса задаётся скорость обмена данными и, если необходимо, включается контроль четности или нечетности.

Для Ethernet необходимо определить режим работы – сервер или клиент. При установке свойства «Сервер (ждем соединения)» включается режим сервера: КС будет ожидать соединение на заданный коммуникационный порт. В противном случае будут производиться периодические попытки установить соединение по определенному адресу TCP (см. ниже) на заданный порт.

При установке свойства «Режим мастера (опрашиваем)» устройства, привязанные к данному порту, являются ПУ. В противном случае – КП. Допускается любая комбинация свойств «Сервер (ждем соединения)» и «Режим мастера (опрашиваем)».

Коммуникационный порт может быть выведен из работы с использованием опции «Выведен из работы».

Измерительный прибор

Для созданного узла коммуникационного порта создаются узлы измерительных приборов (устройств). Является устройство КП  или ПУ определяется свойством коммуникационного порта (см.  «Коммуникационный порт»).

При создании прибора на начальной вкладке «Общие» может потребоваться настройка классификатора; подробная информация о классификации оборудования содержится в документе «Классификация измерительных приборов».

Сетевой адрес. При работе по интерфейсу Ethernet устройству соответствует TCP адрес (может быть задан IP адрес или DNS имя). Для нескольких устройств на одном порту TCP адреса должны совпадать. Не допускается оставлять незаполненный TCP адрес. Если режим работы – сервер, то TCP адрес не используется (обслуживается клиент с любого адреса). Порт TCP в общем случае должен повторять значение, введенное для узла «Коммуникационный порт».

Профиль. Для создания точек измерений, привязываем к прибору созданный ранее профиль. Привязка к профилю осуществляется по нажатию кнопки «Профиль». Для выбранного типа прибора соответствует отдельный профиль. В открывшемся окне необходимо отметить все необходимые точки измерений.

Расширенное руководство по созданию профилей находится в документе «Описание параметров в подсистеме сбора и передачи РСДУ».

Физический адрес. Если описываемое устройство является КП, то физический адрес – это канальный адрес, по которому удаленный ПУ будет обращаться к описываемому КП. Если описываемое устройство является ПУ, то физический адрес – это канальный адрес удаленного КП.

Устройство может быть выведено из работы включением опции «Выведен из работы».

Опции устройства

Свойства конкретного протокола конфигурируются в опциях устройства. При добавлении нового измерителя будут создано минимальное число опций, которые требуются для работы данного типа протокола.

Все доступные опции устройства можно определить, нажав кнопку «Добавить». Тут же будет предложено использовать значение по умолчанию, которое можно отредактировать.

Период опроса устройства в канале (10*мс)

REQDATA_DELAY

Период запроса данных класса 2 при отсутствии на КП данных. Если данные есть, то запросы генерируются немедленно, пока все данные не будут приняты.

Данные класса 1 запрашиваются немедленно при их наличии.

При одном КП на канале имеет смысл установить небольшой период (10-50); при нескольких КП –  период выбирается в зависимости от приоритета запрашиваемых с КП данных. Например, при получении с КП только интегральных значений период можно увеличить.

При запросе ТИ, ТС, особенно при использовании спорадической передачи, период рекомендуется уменьшать.

Период синхронизации времени (с)

TIME_SYNC_ PRESENT

Период посылки команды «Clock synchronization command» (ASDU тип 103), значение определяется в разделе «Необходимые знания» п.7.

Происходит синхронизация времени КП с ПУ. Посылается текущее время хоста КС. Первая синхронизация происходит немедленно при установлении связи с устройством.

Если устройство, описанное в РСДУ5, выступает в качестве КП, данная опция (со значением = 1) позволяет принимать и обрабатывать команду синхронизации времени на хосте. Даже при установленной опции, синхронизация будет пропущена, если хост поддерживает синхронизацию с источника точного времени (команда `ltime -a` возвращает значение «Global Clock is INTP_CLOCK»).

Часовой пояс для синхронизации времени (с)

SYNC_TIME_ZONE_DIFF

Данная опция указывает смещение часового пояса относительно UTC, которое используется при отправке команды синхронизации времени на измерительный прибор или при получении команды синхронизации. Значение опции может быть как положительным, так и отрицательным.

При отсутствии опции в описании прибора, метка времени для синхронизации отправляется/ожидается в локальном часовом поясе.

Пример:

Если прибор ожидает метку времени для синхронизации в часовом поясе UTC+3, то для данной опции необходимо выставить значение 10800.

Если мы ожидаем, что нам будет передана команда синхронизации с меткой времени в часовом поясе UTC-2, то для данной опции необходимо установить значение -7200.

Смещение часового пояса (с)

TIME_ZONE_DIFF

Данная опция указывает на разницу во времени между источником и приёмником: то есть указанное в опции значение прибавляется к метке времени получаемых/передаваемых параметров.

При отсутствии опции в описании прибора метка времени не модифицируется.

Период генерации циклических обновлений (с)

CYCLIC_DATA_ DELAY

Период генерации циклических данных с текущими значениями ТИ, ТС.

Значение определяется в разделе «Необходимые знания» п.8 для МЭК.

Если на момент генерации отправлены не все данные предыдущих периодов, то генерация откладывается. Передача циклических данных прерывается при поступлении данных по выполнению любой команды, спорадическом сообщении и возобновляется по  завершению отправки этих данных.

При установке значения 0, значения ТИ и ТС обновляются спорадически по изменению значения параметра.

Ожидание (T1) устройства на канальном уровне (с)

LINK_TIMEOUT

Время ожидания пакета данных от устройства.

Для МЭК87-0-5-104: тайм-аут Т1 при посылке или тестировании APDU. Для МЭК870-5-101: если пакет не приходит, то последний отправленный на устройство пакет считается не доставленным, и происходит его повторная отправка.

Таймаут (T2) подтверждения при отсутствии данных (с)

Тайм-аут для формирования подтверждения (формата S) в случае отсутствия сообщения с данными Т2 < Т1.

Период (T3) тестирования канала (с)

CHANNEL_TEST_ DELAY

Тайм-аут для посылки блоков тестирования в случае долгого простоя.

Время выполнения команды (с)

Время выполнения команд общего опроса, опроса счетчиков, синхронизации времени. Если за установленное время с КП не приходят все данные по выполнению команды, то при команде:

-         опроса объекты информации, значения которых не обновились, помечаются как некорректные;

-         синхронизации времени посылается сигнал «Внутреннее время измерителя некорректно».

Передатчик прекращает передачу при достижении числа k неподтверждённых APDU формата I.

Формирование подтверждения при приёме (W) кадров

RCV_COUNTER_W

Приёмник передаёт подтверждение по крайней мере после получения w APDU формата I.

Рекомендация: значение w не должно быть более двух третей значения k.

Тип запроса (1 – последовательный, 2 – по изменению)

DEVICE_ QUERY_TYPE

Актуально при настройке протокола IEC60870-5-101. Задает режим опроса измерителей на магистрали: последовательно (значение опции 1), либо по мере необходимости (значение опции 2).

По умолчанию задается тип запроса 2 (по изменению).

Тип кадров: постоянной длины (1), переменной длины (2) , использование обоих типов длины (3)

USE_FRAME_TYPE

Актуально при настройке протокола IEC60870-5-101. Позволяет зафиксировать тип кадра: 1 – постоянной длины, 2 – переменной длины.

По умолчанию задается USE_FRAME_TYPE=3 (использование кадров как с постоянной длиной, так и с переменной длиной).

Запрашивать второй раз состояние канала после его сброса

SECCHAN_STATE_ ASK

Опция актуальна при настройке обмена по протоколу IEC60870-5-101. Позволяет включить или отключить функцию повторного запроса состояния канала после его сброса (0 – функция повторного запроса отключена, 1 – функция повторного запроса включена).

Параметры устройства

Для созданного узла устройства создаётся список параметров.

Каждый параметр привязывается к соответствующему объекту информации, определенному в разделе «Создание и конфигурирование профиля устройства» п.3.5 (следует выбрать точку профиля). В общих свойствах также указывается наименование, класс напряжения и единица измерения.

В параметрах указывается необходимый масштабный коэффициент (по умолчанию установлено значение 1):

ü  для ТИ значение параметра умножается на указанный масштабный коэффициент; при включенном свойстве «Масштабирование» значение параметра будет пересчитываться на основе введенных свойств K, Min и Max;

ü  для ТС при масштабном коэффициенте < 0 происходит инверсия значения.

Значение, введенное в «Приоритет» и «Апертура», влияет на режим передачи данных по протоколам IEC101/104. “Приоритет” используется для указания данных, которые должны передаваться в первую очередь; приоритет задается от 1 (минимальный приоритет) до 255 (максимальный приоритет); по умолчанию (если в настройках указано значение «0») для сигналов телесигнализации используется приоритет 150, для сигналов телеуправления используется приоритет 250, а для остальных – 128. «Апертура» задает минимальную разницу, на которую должно измениться значение параметра, чтобы оно отправилось по спорадическому принципу.

При ненулевом значении «Контроль недостоверности» для данного параметра включается контроль статуса недостоверности: если в течение заданного интервала времени у значения держится  недостоверный статус, то генерируется сигнал о неисправности контролируемого параметра

Если устройство является КП, то необходимо настроить параметр на получение значений от указанного источника (для отправки их на ПУ).

На вкладке «Архивы» можно включить запись архивов для доступных в системе архивных профилей.

 

Резервный канал

Необходимо, чтобы порт, для которого создаётся резервный порт, был сконфигурирован заранее.

Аналогично предыдущим разделам инструкции создаётся и настраивается коммуникационный порт, который будет являться резервным. При этом номер коммуникационного порта – это номер резервного физического порта.

Далее необходимо сменить значение поля «Основной канал» с состояния «Не назначен» на тот порт, который должен быть основным для текущего порта. Не допускается строить иерархию резервных портов, то есть назначать резервным порт, у которого уже есть резерв.

Затем для созданного резервного порта создается измерительный прибор с заполнением IP-порта и IP-адрес резервного канала. Список параметров для данного измерителя не создается, поскольку будет использоваться список параметров с устройства на основном канале.

При загрузке КС начинает вести обмен по основному каналу. Переход на резервный канал КС осуществляет при достижении счетчика сбоев в канале значения RESERVE_FAULTS за время RESERVE_TIME (см. INI параметры КС).

INI параметры сервера КС

SS_USER_ID

ID сервера сигнальной системы, через который будут отправляться сигналы.

TECH_CYCLE

Период генерации запросов данных к сервисам ОИК, выступающих источниками значений параметров на передачу. Задаётся в 10*мс (например, значение 100 соответствует одной секунде).

По умолчанию – значение 100.

DCPCLN_TECH_CYCLE

Период генерации запросов данных к облегченному серверу данных DCP. Задаётся в мс (например, значение 1000 соответствует одной секунде).

По умолчанию – значение 100.

DCP_MASTERSOURCE

Подключаться к основному облегченному серверу данных DCP или к резервному серверу данных DCP.

По умолчанию – значение 1.

AUTO_TUNE_SRC

Автоматическое переключение источников:

0 – отключено;

1 - используется алгоритм переключения источников с возвратом на более приоритетный (в качестве приоритета используется приоритет канала -  DA_SRC_CHANNEL.PRIORITY);

2 – используется алгоритм переключения между источниками данных только при недостоверном состоянии источника (без возврата на самый приоритетный канал).

По умолчанию – значение 0.

INI-параметр используется совместно с MAX_ATTEMPT.

MAX_ATTEMPT

Количество проверок перед переключением с недостоверного источника на резервный источник данных. Используется совместно с AUTO_TUNE_SRC. Проверка выполняется каждый технологический цикл (TECH_CYCLE).

По умолчанию – значение 3.

RESERVE_FAULTS

Количество сбоев в канале для перехода на резервный канал. Если резервный канал отсутствует, то связь с устройством разрывается, и происходят попытки ее восстановить.

Сбоем канала считается: разрыв связи с устройством вследствие внешних причин (неисправность канала, недоступность ресурсов ОС и т.д.), неисправность всех устройств канала (работа хотя бы одного из устройств гарантирует работоспособность канала), ошибка при попытке установить связь с устройством, ошибки в пакете (нарушение CRC, контрольных полей и т.п.).

По умолчанию – значение 1.

RESERVE_TIME

Интервал подсчета сбоев канала. Задаётся в секундах.

Используется совместно с RESERVE_FAULTS (счётчик RESERVE_FAULTS обнуляется по истечении этого времени).

По умолчанию – значение 600.

CHECK_CORRECT

Величина расхождения времени, используется при расчёте достоверизации значений параметров сбора, для которых установлено одно из свойств:  «Проверка на обновление» или «Проверка на изменение». Задаётся в секундах.

По умолчанию – значение 600.

SETTIME_PERIOD

Периодичность коррекции времени на измерителях. Задаётся в минутах.

Функция должна поддерживаться измерителем (поддержка имеется для некоторых моделей СЭТ/СПЕ/ИОН).

По умолчанию – значение 180.

WAIT_TIME

Таймаут сервисных служб. Задаётся в минутах.

Сервис обрывает клиентскую сессию при отсутствии запросов от клиента в течение указанного времени. В случае принудительного обрыва соединения посылается сигнал «Аварийное завершение сессии».

По умолчанию – значение 3.

TRESHOLD_CHAN_ERR

Допустимый порог ошибок в канале. Используется для фиксации ошибок на магистрали и переинициализации драйвера сбора по достижению указанного порога.

По умолчанию – значение 2.

MIN_PERIOD_SENDSIGNAL_QUEUEFULL

Задает периодичность высылки сигнала о переполнении очереди, в которой хранятся изменившиеся значения, предназначенные для передачи по протоколам IEC101/104. Значение измеряется в секундах.

Параметр является необязательным.

Значение по умолчанию – 600 (секунд).

DBLTS_TIMEOUT

Задает продолжительность нахождения двухпозиционного ТС в состоянии «Нейтральный» (после истечения указанного промежутка времени двухпозиционный ТС перейдет из нейтрального состояния в «Неисправность»). Значение измеряется в секундах.

Параметр является необязательным.

Значение по умолчанию – 10 (секунд).
