Требования к наименованию файлов схем и панелей

В наименовании файлов допускаются только английские большие и маленькие буквы, цифры и символ подчёркивания. Ни каких пробелов, запятых, точек (кроме разделения расширения файла), знака тире и пр.

Требования к формированию файлов панелей

1.      Если требуется скрыть подпись к контрольному элементу, то необходимо это делать только сбрасывая флажок «Label Visible»

2.      Нельзя накладывать один контрольный элемент на другой, например, кнопку поверх текста. Исключение – наложение контрольных элементов на картинку (PICTURE). От картинки будет использована только рамка.

3.      Украшения шрифтов (кроме размера и цвета) не поддерживается в текущей версии.

4.      В WEB интерфейсе панелей можно использовать только следующие контрольные элементы CVI:

Требования к формированию файлов схем

1.      Все подписи к элементам схем  необходимо оформлять либо в виде объекта с типом 5 'text', либо в виде диспетчерского наименования. Если необходимо скрыть диспетчерское наименование, то в Модусе надо снять флажок «Подпись к элементу/видимая». Не надо ставить для этой цели цвет подписи равный цвету фона. Правильность типов можно проверить, посмотрев содержимое файла xsde. К типу 5 относится весь текст добавленный в виде объекта, т.е. либо через контекстное меню правой кнопки, либо с помощью элементов библиотеки графических типов.

2.      Для отображения телеизмерений используется тип 134 «прибор цифровой2»

3.      Для конвертации схема должна быть сохранена в формате xsde версии 5.20 (проверить, посмотрев сохранённый файл в любом текстовом редакторе)

4.      Перед сохранением схемы необходимо установить ее размер «Схема»/ «Свойства страницы» /«размер схемы»/ «флажок ‘В ручную’»/ «ОК»

5.      После конвертации расположение элементов на схеме может не полностью совпадать с их расположением на схеме Модус, это связано с особенностями преобразования чисел с плавающей точкой в целые числа. Поэтому рекомендуется при создании схемы в Модус делать привязку элементов к сетке.

После того как схема сделана в редакторе Modus её необходимо сохранить в двух форматах sde и xsde. Описать и настроить в БД РСДУ2 как обычно для формата sde.

 Затем файл формата xsde необходимо сконвертировать в формат .xsde.uni для отображения через WEB клиент.

Конвертирование происходит программой Xsde2Xml.exe расположенной в каталоге \\rsdu2web\utils\Modus

Нажимаем кнопку «Выбрать» выбираем файл и затем «Старт». Протокол работы программы отображается в окне. Результат помещается в файл .xsde.uni который необходимо переписать в каталог \\rsdu2web\schemes

После добавления новых панелей и схем, необходимо в WEB клиенте перечитать конфигурацию нажав CtrlF5 (Обновить кэш)

Описание работы модуля отчетов

1.      Отчеты веб-клиента строятся на основе текстовых конфигурационных файлов в формате «ini» с расширением «wrpt». 

2.      Все файлы отчетов хранятся в папке: КОРНЕВОЙ_КАТАЛОГ_ХОСТА/resource/reports/* (Например /home/www/htdocs/WASUTP/public/resource/reports/*).

3.      При формировании меню веб-клиента (во время первой загрузки в браузере) веб-сервер запрашивает у СУБД информацию о веб-отчетах, находящихся в предпочтениях пользователя, с помощью sql-запроса:

SELECT

    rl.ID ID,

    rl.NAME NAME,

    rl.NAME_RPT FILE_NAME,

    m.ID_PARENT ID_PARENT

FROM us_menu m

    INNER JOIN rpt_lst rl ON rl.id = m.id_param

    INNER JOIN sys_otyp so ON so.id = rl.id_type

WHERE so.define_alias = 'OTYP_WEB_CLIENT_REPORT'

    AND m.id_user = < ID пользователя>

    

4.      Получив ответ, веб-сервер удаляет из него записи с именами файлов, не существующих в папке (информация о несуществующих файлах отображается в меню сообщений приложения, при нажатии на кнопке «Системные сообщения» в правом нижнем углу окна браузера), описанной в пункте 2. На основе оставшихся строится меню «Отчеты».

5.      При нажатии на выбранный пункт меню «Отчеты», браузер посылает AJAX-запрос веб-серверу с указанием id-отчета, веб-сервер формирует данные для построения таблицы отчетов на основе конфигурационного файла .wrpt и отправляет их клиенту в формате JSON.

Структура конфигурационного файла отчета .wrpt

 

Каждый отчет состоит из 4-х основных секций:

1.      Секция описание запросов к СУБД (система работает с СУБД ORACLE и MySQL, выбор СУБД происходит на основе конфигурационного файла wasutp.config.php)

2.      Секция описания полей выбранных записей

3.      Секция описания настроек отображения окна отчета веб-клиента

4.      Секция описания условий, при выполнении которых для соответствующих строк отчета применяются указанные CSS-классы, которые описывается в файле: КОРНЕВОЙ_КАТАЛОГ_ХОСТА/desktop/css/wrpt.css (Например /home/www/htdocs/WASUTP/public/desktop/css/wrpt.css).

 

Пример конфигурационного файла

1.      Секция запросов к СУБД

СУБД ORACLE

Доступные параметры:

Select – описание подзапроса select … from

В этой подсекции описывается предикат select sql-запроса (предикаты where, group by и order запрещены).

Также в этой подсекции описываются позиции для других подсекций (Where, Order, Date), в которые они будут подставлены при компиляции запроса.

Where - описание подзапроса where …

В этой подсекции описываются условия where, которые впоследствии будут объединены оператором AND.

Несколько условий можно объединить в одной подсекции where (например если необходимо использовать оператор OR: where.condition1 = "J_PSTATE.ID_SRCUSER = S_USERS.ID OR J_PSTATE.ID_CAUSE = S_USERS1.ID" )

Order - описание подзапроса order …

В этой подсекции описывается предикат order, который может быть использован для сортировки таблицы отчета по умолчанию. Например, с помощью записи:

«order = "j.dt1970 DESC"» таблица будет отсортирована по убыванию по полю j.dt1970 при открытии соответствующего отчета. При использовании дальнейшей сортировки в окне отчета, условие «order» конфигурационного файла wrpt использоваться не будет.

Date – ключевое слово,  используемое для фильтрации по дате.

В одном отчете не могут одновременно использоваться ключевые слова Where и Date. В этом случае необходимо использовать подзапрос для выборки значений даты:

[sql_mysql]

select = "

    SELECT

        j.dt1970 r_time,

        ss.ID_TYPE SIGN_TYPE,

        ss.NAME SIGN_NAME,

        su.NAME SRCUSER,

        su1.NAME CAUSE,

        dc.NAME DEV_NAME,

        dj.NAME JSTAT_NAME,

        sd.ID DTYP_ID,

        sd.ALIAS DTYP_ALIAS

    FROM (

        SELECT *

        FROM j_dadv

        :date

        ORDER BY dt1970

    ) j

        INNER JOIN sys_sign ss ON ss.ID = j.ID_SIGNAL

        INNER JOIN s_users su ON su.ID = j.ID_SRCUSER

        INNER JOIN s_users su1 ON su1.ID = j.ID_CAUSE

        INNER JOIN da_cat dc ON dc.id = j.id_device

        INNER JOIN da_jstat dj ON dj.ID = j.STATUS

        LEFT JOIN sys_dtyp sd ON sd.id = ss.id_type

  :where

  :order

"

where.condition1 = ""

order = ""

 

 

Пример:

[sql_oracle]

select = "

SELECT

    rownum as RNUM,

    J.dt1970 r_time,

    t.ALIAS sigtype,

    s.NAME signame,

    u.NAME  src_user_name,

    u1.NAME cause_user_name

 FROM (

order = "j.dt1970 DESC"

2.      Секция fields - настройка колонок таблицы пользовательского интерфейса

Доступные опции fields

column – столбец таблицы в sql-запросе

title - обязательное поле - строка в кавычках  (двойных либо одинарных)- заголовок столбца

width - ширина столбца - целочисленное значение без кавычек - необязательное поле

Если поле width не описано или его значение равняется пустой строке (NAME.width = "") – то соответствующего поле будет иметь динамическую ширину.

filter - фильтр для поля, одно из значений: "string", "int", "float", "bool", "combobox" или "date" - необязательное поле.

Если для поля указан фильтр "date", то в окне соответствующего отчета будет отображено 2 поля для ввода «Дата (от)» и «Дата (до)», каждое из которых имеет кнопку в правом углу, при клике по которой появляется календарь, для более удобного поиска даты.

 

Опция column поля, для которого указан фильтр «date», должна описывать столбец в sql-запросе, возвращающий значение типа timestamp (или строкового типа данных, образованного конкатенацией столбца типа timestamp и столбца, содержащего количество микросекунд, например «j.dt1970 + trunc (J.dt_mks/1000000, 3) r_timestamp»).

Если поле filter не описано или его значение равняется пустой строке (NAME.filter = "string") – то для соответствующего поля не будет создан фильтр в интерфейсе пользователя.

sql - sql запрос для формирования combobox поля (поле filter должно быть установлено в «combobox»),

 

Имена полей берутся из секции select ([sql_oracle] или [sql_mysql]).

В качестве имен переменных используются алиасы sql-запроса, например: DATA. column = " J_PSTATE.dt1970", где DATA – алиас колонки  J_PSTATE.dt1970.

3.      Секция window - настройка параметров окна

Доступные параметры:

title - заголовок окна - строка в кавычках - необязательное поле (по умолчанию = названию нункта меню)

width - ширина окна в пикселах - целочисленное значение без кавычек - необязательное поле (выбирается индивидуально в зависимости от разрешения окна браузера)

height - высота окна в пикселах - целочисленное значение без кавычек - необязательное поле(выбирается индивидуально в зависимости от разрешения окна браузера)

size - количество строк в таблице отчета - целочисленное значение без кавычек - необязательное поле (если строки таблицы не влезут в окно с заданной высотой, появится скроллинг)

updatePeriod – период времени для обновления таблицы в пользовательском интерфейсе в миллисекундах - целочисленное значение без кавычек.

Пример:

[window]

id = "statement_processes"

title = "Журнал состояния процессов"

width = 1100

height = 650

size = 26

updatePeriod = 5000

 

4.      Секция условий

Доступные параметры

cls – CSS-класс строки, который описывается в файле: КОРНЕВОЙ_КАТАЛОГ_ХОСТА/desktop/css/styles.css (Например /home/www/htdocs/WASUTP/public/desktop/css/styles.css).

column – алиас поля в SQL-запросе

condition.greater – условие “ > “

condition.less  – условие “ < “

condition.equal  – условие “ = “

condition.equalgreater – условие “ >= “

condition.equalless – условие “ <= “

Пример:

condition1.column = 'SIGNAME'

condition1.condition.equal = 'Включение'

 

condition2.cls = 'green-text'

condition2.column = 'SIGNAME'

    LEFT JOIN sys_sign s ON s.ID = j.id_signal

    :where

    :order
