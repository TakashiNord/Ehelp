Сборка и установка серверных модулей iSMS из исходных файлов

Сборка и установка серверных модулей iSMS

Перед выполнением сборки и установки приложений нужно обязательно убедиться в том, что имеется актуальный конфигурационный файл rsdu.config, в котором могут быть прописаны индивидуальные параметры сборки приложений. Если таковой файл обнаруживается, его нужно заменить после распаковки архива.

1.   Распаковать архив iSMS_linuxRT_Revision_DD-MM-YY_HH-mm-ss.tar.bz2

Пример:

$> tar -jxvf ./iSMS_linuxRT_Revision_DD-MM-YY_HH-mm-ss.tar.bz2

2.   Зайти в каталог с исходниками  iSMS_linuxRT_Revision_DD-MM-YY_HH-mm-ss

Пример:

$> cd ./iSMS_linuxRT_Revision_DD-MM-YY_HH-mm-ss

3.   Проверить файл конфигурации сборки ‘rsdu.config’ в случае необходимости отредактировать.

4.   Выполнить скрипт конфигурирования проектов configure. ВНИМАНИЕ! Выполнение скрипта configure обязательно перед первой сборкой. Он генерирует необходимые для сборки файлы.

Пример:

$> ./configure

5.   Запустить сборку проекта через Makefile.

Пример:

$> make

6.   Выполнить установку собранных серверов. ВНИМАНИЕ! Требуются права root.

Пример:

$> sudo make install

7.   В процессе установки будут заданы вопросы о необходимости выполнения дополнительных действий.

Do You want to make full installation? – Требуется ли выполнять полную установку или только обновление модулей.

Do You want to create needed directories? – Требуется ли создавать все необходимые директории.

Do You need to install sqlite3 library? – Устанавливаеть или нет библиотеку sqlite3.

Do You need to install 'ema' script? – Устанавливать или нет скрипт ema.

Set time delay waiting for oracle. [Sec] (400) – Установить время ожидания загрузки СУБД Oracle перед запуском всех серверов РВ.

Do You need to install 'ema_autoload' script? – Устанавливать или нет скрипт ema_autoload.

Do You need add 'ema_autoload' script to autolad? – Добавлять или нет скрипт ema_autoload в автозагрузку.

Do You need to install default ema.cfg file? – Устанавливать или нет заготовку конфигурационного файла ema.cfg.

Do You need to install default host.ini file? – устанавливать или нет заготовку конфигурационного файла host.ini.

Do You need to install 'ema_time.sh' script and its configuration file? – Устанавливать или нет скрипт контроля времени.

Do You need to add 'ema_time.sh' to cron? – Добавлять или нет запуск скрипта контроля времени по рассписанию.

Do You need to modify 'sysctl_conf'? – Исправлять или нет рекомендуемые системные параметры.

8.   После установки необходимо отредактировать файлы /etc/ema/ema.cfg и /etc/ema/host.ini (в случае использования функции контроля системного времени на хосте необходимо также подготовить конфигурационный файл /etc/ema/ema_time.cfg).

Если перечисленные файлы отсутствуют, то в качестве образца можно использовать шаблонные  файлы /etc/ema/ema.cfg.default и /etc/ema/host.ini.default, а также /etc/ema/ema_time.cfg.default. Их надо переименовать в ema.cfg, host.ini и /etc/ema/ema_time.cfg соответственно и затем отредактировать.

Дополнительные возможности скрипта configure

Дополнительно на этапе конфигурирования с помощью скрипта configure, можно указать дополнительные параметры. О дополнительных параметрах можно узнать, выполнив команду configure --help .

Возможные параметры:

Ключ

Описание

--prefix=DIR

Указывает путь для установки. Значение по умолчанию /usr. Исполняемые файлы по умолчанию будут установлены  в /usr/bin. Библиотеки по умолчанию будут установлены в /usr/lib. Файлы заголовков по умолчанию будут установлены в /usr/include.

--prefix-bin=DIR

Указывает путь для установки исполняемых файлов. По умолчанию <PREFIX>/bin

--prefix-lib=DIR

Указывает путь для установки библиотек. По умолчанию <PREFIX>/lib

--prefix-inc=DIR

Указывает путь для установки заголовочных файлов. По умолчанию <PREFIX>/include

--enable-<Module>[=yes|no]

Указывает включать или нет проект в сборку. В качестве <Module> можно указывать: all, libs, others, servers

--disable-<Module>

Отключает проект из сборки

--enable-all

Включает все проекты в сборку (за исключением проектов, отключённых с помощью ключей)

--disable-all

Отключает все проекты из сборки (за исключением проектов, включённых с помощью ключей)

--rsdu-confog

Указывает путь к конфигурационному файлу

--rsdu-config-out

Указывает куда вывести результирующий конфигурационный файл.

--defines-<Module>=Define1,Define2

Добавляет дополнительные объявления для сборки определённой версии модуля

--rfcopy-<Module>=Copy1,Copy2

Создаёт копии модулей

--install-libsqlite3[=(yes|no)]

Значение по умолчанию устанавливать или нет libsqlite3

--install-ema-script[=(yes|no)]

Значение по умолчанию устанавливать или нет скрипт ‘ema’

--install-ema-autoload-script[=(yes|no)]

Значение по умолчанию устанавливать или нет скрипт автозагрузки ‘ema_autoload’

--install-ema-cfg-file[=(yes|no)]

Значение по умолчанию устанавливать или нет заготовку конфигурационного файла ‘ema.cfg.default’

--install-host-ini-file[=(yes|no)]

Значение по умолчанию устанавливать или нет заготовку конфигурационного файла ‘host.ini.default’

--install-ema-time-script[=(yes|no)]

Значение по умолчанию устанавливать или нет скрипт контроля времени ‘ema_time.sh’

--update-sysctl-conf[=(yes|no)]

Прописывать или нет в конфигурационный файл ‘sysctl.conf’ значений требуемых для iSMS

--wait-oracle-delay[=<seconds>]

Время ожидания СУБД Oracle перед запуском серверов iSMS. Секунд.

Скрипт configure при старте пытается получить настройки из конфигурационного файла по умолчанию – rsdu.config.

По завершению работы скрипта configure текущий файл rsdu.config обновляется и может иметь другое содержание, если в первоначальном файле были заданы некорректные параметры.

Описание

Собрать только утилиты.

Подготовка архива с исходниками iSMS

Для создания архива с исходниками достаточно в корневом каталоге выполнить команду make dist. Перед выполнением этой команды обязательно должен быть выполнен скрипт configure хотя бы 1 раз.

Пример:

$> ./configure

$> make dist

Замечание. После команды configure (непосредственно перед созданием архива сборки) рекомендуется также выполнять команду make dist-clean.

В результате, в текущем каталоге будет создан файл вида iSMS_linuxRT_Revision_DATE_TIME.tar.bz2, который будет содержать все необходимые для сборки серверов файлы.

Обратите внимание, что сборочный архив будет подготовлен согласно конфигурационному файлу rsdu.config.

Приложение 1. Структура файла rsdu.config

#Секция общих настроек

[common]

prefix=/usr

prefix_bin=

prefix_lib=

prefix_inc=

inst_libsqlite3=yes

inst_ema_script=yes

inst_ema_autoload_script=

inst_ema_cfg_file=

inst_host_ini_file=

inst_ema_time_script=

update_sysctl_conf=

[modules]

smagent=yes

sysmon=yes

#Секция дополнительных параметров для сборки модуля

[smagent]

copyexec=agent1,agent2

 

[elreg]

defines=

copyexec=

mkcalc_out_file=calc_el.cfg

mkcalc_server=elreg

mkcalc_db_login=rsduadmin

mkcalc_db_password=passme

mkcalc_srcdir=.

mkcalc_samba=no

mkcalc_smb_username=admin

mkcalc_smb_password=passme

mkcalc_smb_sname=//RESOURCE/Resource/src

mkcalc_fsrcdir=/root/LinuxRT/Formula

enmacsrv

Использовать реализация для работы с Enmac PowerOn.

EMA

layout

Влияет на формат времени при запросе макета из БД. При включенном параметре, формат времени – "%04d-%02d-%02d %02d:%02d:%02d". Иначе формат времени – "%02d-%02d-%04d".

PID

rtarch

Для подготовки специальной версии модуля RTARCH в системе АРЧМ. При включенном параметре архивные параметры читаются из ARC_VLIST_PID (по умолчанию список архивных параметров читается из ARC_VLIST_RT).

RF

rtarch

Используется для подготовки FAKE-версии модуля rtarch: модуль принимает архивные данные от технологических серверов, но архивные файлы не обновляет.

schit40

Определение для включения внешнего источника статуса квитирования. По умолчанию включено (в заголовочном файле schit40.h)!

Для подготовки специальной версии модуля tmdap, который работает в режиме RESERVE (никогда не переходит в состояние MASTER).
